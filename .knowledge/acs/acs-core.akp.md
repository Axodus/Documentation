## 1. ACS Identity

acs_expanded_name:
  - AI_Coordination_System

acs_role:
  primary:
    - operational_intelligence_layer
    - AI_assisted_coordination_layer
    - agentic_analysis_runtime
    - business_request_support_layer
    - governance_review_support_layer
    - documentation_and_knowledge_support_layer
    - risk_classification_assistant
    - ecosystem_memory_interface

acs_is_not:
  - unrestricted_autonomous_executor
  - replacement_for_governance
  - replacement_for_human_accountability
  - replacement_for_security_audit
  - replacement_for_legal_or_compliance_review
  - treasury_controller_without_governance
  - smart_contract_deployer_without_review
  - magic_decision_oracle
  - marketing_chatbot_only

acs_thesis:
  - Axodus is a modular ecosystem with multiple nuclei, governance layers, financial flows, token utility, trading operations, academy rewards, marketplace commerce, and custom DAO requests.
  - This complexity requires a persistent intelligence layer capable of analyzing context, detecting risk, summarizing decisions, producing documentation, and supporting operators.
  - ACS provides that intelligence layer through specialized agents, structured runtimes, semantic memory, and tool-enabled workflows.
  - ACS must remain auditable, permissioned, and accountable.
  - ACS should help humans and governance make better decisions, not remove them from the loop.

## 11. Public Documentation Guidance

When creating public ACS docs:

- Explain ACS as AI-assisted coordination.
- Define its role across Governance, Business, Trading, Academy, Treasury, Marketplace, and Accountability.
- Define agent roles clearly.
- Explain runtime lifecycle.
- Explain tools/MCPs as permissioned capability surfaces.
- Explain human oversight.
- Include strong risk and security language.
- Avoid claiming full autonomy unless a capability is implemented and approved.
