## Purpose

This folder contains the deep semantic knowledge base for the ACS nucleus of Axodus.

ACS stands for AI Coordination System. It is the intelligence, analysis, orchestration, and operational support layer of the Axodus ecosystem.

These files are not public-facing documentation pages. They are agent-oriented knowledge packs designed to help coding agents, documentation agents, governance agents, ACS runtime agents, and future MCP systems generate accurate and consistent public documentation and implementation plans.

## ACS Thesis

ACS is not a chatbot layer.

ACS is an AI-assisted operational coordination layer that can support:

- Business request intake analysis;
- governance proposal analysis;
- risk classification;
- documentation generation;
- technical scoping;
- treasury and trading review support;
- product planning;
- agent-based monitoring;
- knowledge assimilation;
- operational summaries;
- adversarial review;
- structured handoff to human operators and governance layers.

ACS must never be documented as an unrestricted autonomous authority.

ACS supports governance, Business, product nuclei, and human operators. It does not replace human accountability, security review, treasury supervision, smart contract audits, or governance authority.

## Core Agent Archetypes

The current ACS identity model includes three major agent archetypes:

- Morpheus: strategic, narrative, governance alignment, ecosystem coherence.
- Trinity: trading, operations, execution intelligence, P&L/risk monitoring.
- Agent Smith: adversarial intelligence, stress testing, abuse resistance, unsafe-access friction.

These archetypes may be expanded over time, but all agents must operate under explicit authority boundaries.

## How Agents Should Use This Pack

Agents must use this pack as contextual source material when generating or reviewing ACS documentation.

When generating public pages, agents should:

- write final documentation in English;
- describe ACS as an AI-assisted coordination layer;
- avoid presenting agents as final authority;
- clearly define agent roles and limits;
- define runtime phases from request intake to review and handoff;
- explain human and governance validation requirements;
- include security, privacy, and operational risks;
- explain MCP/tool usage as capability surfaces that require permissions and logs;
- connect ACS to Governance, Business, Trading, Treasury, Academy, Marketplace, and Accountability;
- distinguish active, planned, experimental, and conceptual capabilities.

## Non-Negotiable ACS Invariants

- ACS is advisory and assistive unless explicitly authorized by governance-approved workflows.
- ACS must not have unrestricted treasury execution authority.
- ACS must not have unrestricted smart contract deployment or upgrade authority.
- ACS must not bypass human review for sensitive decisions.
- ACS must not replace governance.
- ACS outputs require traceability when used for material decisions.
- ACS tools and MCPs must be permissioned, logged, scoped, and revocable.
- ACS should support knowledge continuity through semantic memory and documentation alignment.
- ACS must be designed for auditability, not opaque automation.
- ACS should reduce operational risk, not create unaccountable execution risk.
