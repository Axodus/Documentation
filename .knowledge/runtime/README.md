## Purpose

This folder contains the deep semantic knowledge base for the Axodus Runtime nucleus.

These files are not public-facing documentation pages. They are agent-oriented knowledge packs designed to help coding agents, documentation agents, ACS agents, governance reviewers, product planners, delivery operators, and future MCP systems generate accurate and consistent public documentation and implementation plans.

Runtime is the operational protocol that defines how work moves through Axodus from request to completion.

## Runtime Thesis

Axodus Runtime is not a single application screen.

Runtime is the cross-nucleus operational lifecycle that gives every request, proposal, delivery, review, validation, delay, change, escalation, and accountability record a clear beginning, middle, and end.

Runtime is needed because Axodus includes:

- Governance decisions;
- Business service requests;
- Academy courses and reward policies;
- Trading strategy access and risk workflows;
- Treasury actions;
- ACS-assisted analysis;
- Marketplace listings and commerce flows;
- Tokenomics and locked reward rules;
- DAO federation and plugin requests;
- BBA campaigns;
- Coder handoffs and implementation tasks.

Without Runtime, the ecosystem becomes informal, hard to audit, and vulnerable to scope creep, unclear authority, missed approvals, silent delays, and undocumented changes.

## Runtime Mission

Runtime provides the shared operational structure that ensures:

- requests are captured;
- context is collected;
- risk is classified;
- responsible nuclei are identified;
- ACS analysis is used safely;
- human review is performed where required;
- governance escalation happens when needed;
- scope is defined;
- milestones are tracked;
- confirmations and approvals are recorded;
- delays are explained transparently;
- change requests are controlled;
- delivery is validated;
- acceptance is documented;
- execution receipts and accountability records are preserved.

## Non-Negotiable Runtime Invariants

- Every material request must have a clear status.
- Every material request must have an owner or responsible nucleus.
- Intake is not approval.
- ACS analysis is not final authority.
- Scope must exist before delivery commitment.
- Governance-sensitive requests must be escalated.
- Change requests must be recorded and classified.
- Delays must be explained with reason, impact, and next action.
- Delivery must be validated against acceptance criteria.
- Sensitive execution requires review and recordkeeping.
- Accountability records must exist for material decisions and executions.
- Runtime must protect transparency, not create bureaucracy without purpose.
- Public Runtime documentation must be written in English.
